Rendersend — Product Requirements

Vision:
A hosting platform for HTML artifacts (reports, dashboards, analyses) generated by Claude or any LLM, shared via private links with ACL-based access control and zero-access encryption. Distribution happens through an MCP server that any Claude user can connect to.

Target Users:
Finance teams sharing reconciliation reports, financial analyses, and dashboards
Accountants distributing audit summaries and client-facing outputs
Analysts and consultants sharing data analyses and findings
Any Claude user who wants to share a generated HTML artifact securely

Core User Flow:
User asks Claude (or another LLM) to generate an HTML report or dashboard
User says "share this" → triggers the Rendersend MCP tool 
   Or, User can download HTML and then drag and drop or upload after visiting our site
MCP receives HTML + recipient list, encrypts content, returns a clickable link
Owner sends the link via email, Slack, etc.
Recipient clicks link, authenticates, content decrypts client-side and renders in a sandboxed iframe

Functional Requirements
Publishing:
Accept arbitrary HTML content via MCP share_html(content, acl, options) call OR user uploads it from our website
Return a clickable shareable URL within seconds
Support optional metadata: title, description, expiry, view limits
Support content updates by the owner (versioning) without breaking existing links
Allow owner to revoke access at any time

Access Control:
Role-based ACLs: Owner, Editor, Viewer
Per-recipient access via verified identity (email-based at minimum)
Owner can add or remove recipients after publishing
Owner sees audit log: who viewed, when, from where (IP/region only, not full PII)
Optional public-link mode (anyone with link) for non-sensitive content
Optional password-protected mode as a lightweight alternative to full ACL

Encryption & Security:
Zero-access encryption: server cannot read content after the moment of ingestion
Content encrypted with AES-GCM using a per-document content key
Content key wrapped per-recipient using their public key (RSA-OAEP or X25519)
Each user has a client-generated keypair; private key wrapped by user passphrase or device key
Content signed by owner for integrity verification on the viewer side
Strict CSP and sandboxed iframe rendering to isolate published HTML from the host platform
HTTPS-only, HSTS, no plaintext logging of request bodies on the MCP ingress
Roadmap: optional local MCP for true end-to-end encryption (encryption happens on user's machine, server never sees plaintext)

Rendering:
Sandboxed iframe with restrictive CSP for all published HTML
Block or proxy external resource loads to prevent metadata leakage and tracking
Support for inlined assets (JS, CSS, fonts, images) within the encrypted blob
Mobile-responsive viewer
Print-friendly view for finance/audit users who need PDF exports

MCP Server:
Hosted MCP endpoint that any Claude user can connect to via OAuth
Tools exposed: share_html, update_html, revoke_access, list_my_shares, get_share_acl
OAuth-based authentication tying MCP calls to user accounts
Rate limits per user to prevent abuse
Roadmap: local MCP package (npm/binary) for users requiring true E2EE

Account & Identity:
Email-based account signup with optional SSO (Google, Microsoft) for the finance audience
Client-generated keypair on first login; private key encrypted with passphrase
Key recovery mechanism (recovery code or trusted-device escrow) — must not compromise zero-access property
Organization/team accounts with shared workspaces (post-v1)

Lifecycle Management:
Configurable expiry (24h, 7d, 30d, never)
Optional view-count limits (e.g., burn after first view)
Owner-initiated deletion is immediate and cryptographically enforced (delete the wrapped keys)
Soft-delete window (e.g., 7 days) before permanent ciphertext deletion

Audit & Compliance:
Owner-visible access log per share: timestamp, recipient identity, action (view/download)
Exportable audit log for compliance use cases
SOC 2 Type II as a roadmap target for finance/enterprise customers
GDPR-compliant data handling, including right-to-deletion

Non-Functional Requirements:
Link generation latency under 2 seconds for typical HTML payloads (under 1 MB)
Viewer page load under 3 seconds on standard broadband
Support HTML payloads up to 10 MB in v1, larger via roadmap
99.9% uptime SLA target
Multi-region storage for low-latency global access

Explicit Non-Goals (v1):
Real-time collaboration or multi-user editing
Comments, reactions, or social features on shared artifacts
Native PDF or DOCX hosting (HTML only in v1)
Custom domain hosting per customer (post-v1)
Mobile apps (web-only in v1)

Trust & Positioning:
Honest marketing: "zero-access encrypted hosting" for the hosted MCP path; reserve "end-to-end encrypted" for the local MCP path only
Public security documentation explaining the threat model and what is and isn't protected
Open-source the client-side encryption and rendering code to enable third-party verification

Open Questions to Resolve Before Build:
Identity model details: email-only, SSO-required, or pluggable
Key recovery UX: recovery code vs. trusted-device vs. social recovery
External resource policy: strict block, proxy-and-cache, or inline-only
Pricing model: free tier limits, paid tier features, team vs. individual
MCP authentication: OAuth flow specifics and token lifecycle
Whether the MVP should ship hosted-only or hosted + local MCP simultaneously
